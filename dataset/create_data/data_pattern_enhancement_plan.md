# 데이터 패턴 보강 계획

이 문서는 현재 `phase1_acquisition.py` ~ `phase4_ml_prep.py` 데이터 생성 흐름에 추가하면 좋은 패턴과, 이를 어떤 방식으로 반영할지 정리한 계획서입니다.

목표는 모델이 더 쉽게 학습하도록 데이터를 인위적으로 과하게 만들기보다, 대학 물품관리 업무에서 자연스럽게 발생할 수 있는 신호를 합성 데이터에 추가하는 것입니다.

## 1. 보강 범위

### 1.1. 이번에 반영할 패턴

이번 보강에서는 다음 패턴만 반영합니다.

| 패턴 | 반영 여부 | 이유 |
| --- | --- | --- |
| 점검/수리 이력 | 반영 | 고장/노후화 예측에 직접적인 선행 신호가 됨 |
| 사용량/가동률 | 반영 | 같은 품목이라도 실제 사용 강도 차이를 표현할 수 있음 |
| 구매 배치/코호트 | 반영 | 같은 시기 대량 구매 장비의 동시 교체 수요를 표현할 수 있음 |
| 학사/예산 일정 | 반영 | 월별 수요 예측과 조달계획의 계절성을 설명할 수 있음 |
| 부서별 예산/관리 성향 | 반영 | 부서마다 교체 정책이 다르다는 현실적인 차이를 표현할 수 있음 |

### 1.2. 이번에 제외할 패턴

다음 패턴은 이번 보강에서 제외합니다.

| 패턴 | 제외 이유 |
| --- | --- |
| 설치 장소/환경 패턴 | 장소 데이터까지 만들면 합성 데이터 설계가 지나치게 복잡해지고, 실제 시스템 컬럼과 멀어질 수 있음 |
| 모델/제조사/품질 등급 패턴 | 제조사/모델 품질을 임의로 넣으면 근거 설명이 어려워지고 합성 데이터 티가 강해질 수 있음 |

## 2. 전체 설계 원칙

### 2.1. 데이터 누수 방지

모델 학습에 사용할 수 있는 feature는 예측 시점에 실제로 알 수 있는 정보여야 합니다.

따라서 다음 정보는 feature로 직접 넣지 않습니다.

- 실제 불용확정일자
- 실제 처분확정일자
- 실제수명
- 실제 고장/불용 결과 이후에만 알 수 있는 정보

반대로 다음 정보는 예측 전에 알 수 있거나 운영 중 누적되는 정보이므로 feature로 사용할 수 있습니다.

- 취득일자
- 취득금액
- 내용연수
- 운용부서
- 사용 강도 추정치
- 수리/점검 누적 횟수
- 구매 배치 규모
- 부서별 관리 성향
- 학사/예산 시즌 여부

### 2.2. 합성 데이터 티를 줄이는 방향

패턴을 너무 강하게 넣으면 모델 성능은 좋아질 수 있지만, 교수님 관점에서 “정답을 너무 쉽게 심었다”는 인상을 줄 수 있습니다.

따라서 보강 값은 다음처럼 설계합니다.

- 품목, 부서, 가격, 운용연차에 따라 확률적으로 발생하도록 한다.
- 항상 같은 결과가 나오지 않도록 적당한 랜덤성을 둔다.
- 하나의 feature만으로 정답이 결정되지 않도록 여러 신호가 약하게 겹치게 한다.
- 결과 변수와 직접적으로 동일한 값을 넣지 않는다.

## 3. Phase별 변경 계획

## 3.1. Phase 1: 취득 데이터 생성 보강

대상 파일:

- `dataset/create_data/phase1_acquisition.py`

Phase 1에서는 구매 시점에 이미 알 수 있는 정보를 추가합니다.

### 3.1.1. 구매 배치 ID 추가

현재 Phase 1은 한 번의 구매 건에 `수량`을 넣고, Phase 2에서 이를 개별 자산으로 분화합니다.  
이 구매 건 단위가 곧 `동일 배치`입니다.

추가할 컬럼:

| 컬럼명 | 의미 |
| --- | --- |
| `구매배치ID` | 같은 구매 건에서 나온 자산을 묶는 ID |
| `대량구매여부` | 수량이 일정 기준 이상인지 여부 |
| `구매배치수량` | 해당 구매 건의 수량 |

생성 방식:

```text
구매배치ID = B + 취득연도 + 순번
대량구매여부 = 수량 >= 10이면 1, 아니면 0
구매배치수량 = 현재 취득 행의 수량
```

예시:

```text
B202000123
```

기대 효과:

- 같은 시기 구매된 노트북/PC/가구가 비슷한 시기에 교체되는 패턴을 모델이 배울 수 있음
- 월별 조달 수요가 배치 단위로 묶여 발생하는 현상을 설명 가능

### 3.1.2. 부서별 예산/관리 성향 마스터 추가

부서마다 물품을 빨리 교체하는 곳과 오래 쓰는 곳이 있을 수 있습니다.  
이를 너무 세밀한 실제 예산처럼 만들지는 않고, 낮음/보통/높음 수준의 성향값으로 넣습니다.

추가할 컬럼:

| 컬럼명 | 의미 |
| --- | --- |
| `부서예산등급` | 부서의 예산 여력 등급 |
| `부서교체성향` | 물품을 빠르게 교체하는 성향 |
| `부서관리성향` | 물품을 점검/관리하는 성향 |

값 범위:

| 컬럼명 | 값 |
| --- | --- |
| `부서예산등급` | `LOW`, `MEDIUM`, `HIGH` |
| `부서교체성향` | 0.9 ~ 1.15 |
| `부서관리성향` | 0.9 ~ 1.1 |

해석:

- `부서교체성향`이 높으면 같은 상태라도 교체가 조금 빨라질 수 있음
- `부서관리성향`이 높으면 점검/수리 이력이 더 잘 누적되고, 수명이 약간 늘어날 수 있음

주의:

이 값은 실제 예산 금액이 아니라 “부서 운영 성향을 표현하는 합성 지표”로 설명합니다.

## 3.2. Phase 2: 생애주기 시뮬레이션 보강

대상 파일:

- `dataset/create_data/phase2_lifecycle.py`

Phase 2에서는 실제 운용 중 발생하는 누적 신호를 생성합니다.

### 3.2.1. 사용량/가동률 패턴 추가

같은 노트북이라도 실습실 공용 장비와 행정실 개인 장비는 사용 강도가 다릅니다.  
이를 자산별 feature로 만들기 위해 사용량 관련 값을 생성합니다.

추가할 컬럼:

| 컬럼명 | 의미 |
| --- | --- |
| `월평균사용시간` | 한 달 평균 사용 시간 |
| `주당사용일수` | 일주일 평균 사용일 |
| `공용장비여부` | 여러 사용자가 함께 쓰는 장비인지 |
| `수업사용여부` | 수업/실습에 쓰이는 장비인지 |
| `사용강도지수` | 위 값을 조합한 최종 사용 강도 |

생성 방식:

품목군과 부서 유형을 함께 반영합니다.

| 조건 | 사용량 패턴 |
| --- | --- |
| SW/공학/전산 계열 + PC/모니터 | 사용시간 높음, 수업사용여부 높음 |
| 행정부서 + 프린터/스캐너 | 사용시간 중간 |
| 가구류 | 사용시간 대신 공용장비여부 중심 |
| 서버/네트워크 장비 | 월평균사용시간 높음, 주당사용일수 높음 |

예시 공식:

```text
사용강도지수
= 부서가혹도
  * 품목사용계수
  * 공용장비계수
  * 수업사용계수
  * 랜덤보정
```

기대 효과:

- 현재 `부서가혹도`보다 더 구체적인 사용 강도 신호가 생김
- 모델이 “오래된 장비”뿐 아니라 “많이 사용된 장비”를 위험하게 볼 수 있음

### 3.2.2. 사용강도를 실제 수명에 약하게 반영

사용강도지수는 단순 feature로만 두지 않고, Phase 2의 실제 수명 생성에도 약하게 반영합니다.

현재 수명 생성 방식:

```text
최종 수명
= 기본 샘플 수명
  * (1 / 부서가혹도)
  * 가격 보정
  * 관리상태 보정
  * 사용환경 랜덤 보정
```

변경 후:

```text
최종 수명
= 기본 샘플 수명
  * (1 / 부서가혹도)
  * (1 / 사용강도보정)
  * 가격 보정
  * 관리상태 보정
  * 부서관리성향보정
  * 랜덤 보정
```

단, `사용강도보정`은 너무 강하게 주지 않습니다.

권장 범위:

```text
사용강도보정 = 0.9 ~ 1.25
```

높은 사용강도는 수명을 짧게 만들지만, 이것 하나만으로 고장일이 결정되지는 않도록 합니다.

### 3.2.3. 점검/수리 이력 생성

실제 자산은 고장 직전까지 아무 신호가 없다기보다, 중간에 점검이나 수리 이력이 쌓이는 경우가 많습니다.

새 출력 파일:

| 파일 | 의미 |
| --- | --- |
| `04_04_maintenance_list.csv` | 점검/수리 이력 |

생성할 컬럼:

| 컬럼명 | 의미 |
| --- | --- |
| `점검수리일자` | 점검 또는 수리 발생일 |
| `물품고유번호` | 대상 자산 |
| `G2B_목록번호` | 품목 코드 |
| `G2B_목록명` | 품목명 |
| `운용부서` | 당시 운용부서 |
| `점검수리구분` | 정기점검, 장애점검, 수리, 부품교체 |
| `처리결과` | 정상, 경미수리, 주요수리, 교체권고 |
| `수리비용` | 수리/부품교체 비용 |
| `장애심각도` | 0~3 등급 |
| `비고` | 간단한 설명 |

생성 원리:

| 조건 | 점검/수리 발생 확률 |
| --- | --- |
| 운용연차가 낮음 | 정기점검 위주, 수리 적음 |
| 내용연수에 가까워짐 | 장애점검/수리 증가 |
| 내용연수 초과 | 교체권고 확률 증가 |
| 사용강도지수 높음 | 수리 횟수 증가 |
| 부서관리성향 높음 | 정기점검 횟수 증가 |

점검/수리 이벤트는 취득일자 이후, 기준일 또는 불용일자 이전에만 생성합니다.

중요:

수리 이력은 고장 결과를 그대로 복사하면 안 됩니다.  
따라서 “불용되기 직전에 무조건 교체권고”가 아니라, 확률적으로 수리/점검이 늘어나도록 만듭니다.

### 3.2.4. 수리비용 생성

수리비용은 품목 가격과 장애심각도에 따라 생성합니다.

예시:

```text
정기점검: 0원 또는 소액
경미수리: 취득금액의 1~3%
주요수리: 취득금액의 5~12%
부품교체: 취득금액의 8~18%
```

상한:

```text
수리비용 <= 취득금액 * 0.3
```

이렇게 하면 `취득금액대비수리비율` 같은 feature를 만들 수 있습니다.

### 3.2.5. 학사/예산 일정 패턴 유지 및 보강

현재 Phase 2에는 불용/처분 날짜를 2월, 8월, 12월로 몰아주는 로직이 있습니다.

이를 유지하면서, 추후 Phase 4에서 사용할 수 있도록 불용/처분 이벤트 날짜 기반 파생 feature를 만들 계획입니다.

다만 모델 입력에는 실제 불용/처분월을 그대로 넣지 않고, 예측 기준일 또는 분석 대상 기간의 월/학기 정보를 사용해야 합니다.

## 3.4. Phase 4: AI 학습용 데이터 보강

대상 파일:

- `dataset/create_data/phase4_ml_prep.py`

Phase 4에서는 Phase 1/2에서 추가된 원천 컬럼과 점검/수리 이력을 학습 feature로 요약합니다.

### 3.4.1. 점검/수리 이력 요약 feature

`04_04_maintenance_list.csv`를 자산별로 집계해 `phase4_training_data.csv`에 붙입니다.

추가할 feature:

| 컬럼명 | 의미 |
| --- | --- |
| `누적점검수리횟수` | 전체 점검/수리 횟수 |
| `누적수리횟수` | 수리/부품교체/주요수리 횟수 |
| `최근1년수리횟수` | 기준일 이전 1년 내 수리 횟수 |
| `최근2년수리횟수` | 기준일 이전 2년 내 수리 횟수 |
| `마지막수리후경과개월` | 마지막 수리일 이후 경과 개월 |
| `누적수리비용` | 총 수리비용 |
| `취득금액대비수리비율` | 누적수리비용 / 취득금액 |
| `최대장애심각도` | 발생한 장애 중 가장 높은 심각도 |
| `교체권고횟수` | 처리결과가 교체권고인 횟수 |

데이터 누수 방지:

각 자산별 기준일 이전의 점검/수리 이력만 집계합니다.

```text
집계 대상 점검수리일자 <= 기준일
```

여기서 기준일은 Phase 4에서 이미 계산하는 `기준일`입니다.

### 3.4.2. 사용량/가동률 feature

Phase 2에서 생성한 자산별 사용량 컬럼을 Phase 4에 그대로 넘깁니다.

추가할 feature:

| 컬럼명 | 의미 |
| --- | --- |
| `월평균사용시간` | 월 단위 평균 사용량 |
| `주당사용일수` | 주 단위 사용 빈도 |
| `공용장비여부` | 공용 장비 여부 |
| `수업사용여부` | 수업/실습 사용 여부 |
| `사용강도지수` | 최종 사용 강도 |

기대 효과:

- 같은 연차의 자산이라도 사용량이 많은 장비의 위험도를 높게 평가 가능
- 부서명 기반 가혹도보다 더 직접적인 설명 가능

### 3.4.3. 구매 배치/코호트 feature

Phase 1에서 만든 구매 배치 정보를 Phase 2를 거쳐 Phase 4까지 유지합니다.

추가할 feature:

| 컬럼명 | 의미 |
| --- | --- |
| `구매배치ID` | 동일 구매 건 식별자 |
| `구매배치수량` | 같은 구매 건의 수량 |
| `대량구매여부` | 대량 구매 여부 |
| `동일배치운용연차평균` | 같은 배치 자산들의 평균 운용연차 |
| `동일배치자산수` | 같은 배치에 속한 자산 수 |

주의:

`동일배치처분율`, `동일배치불용률`처럼 결과 이후 정보를 포함하는 값은 기본 feature로 넣지 않습니다.  
이 값은 모델 평가용 분석에는 쓸 수 있지만, 예측 feature로는 누수 위험이 있습니다.

### 3.4.4. 학사/예산 일정 feature

모델이 월별 수요 패턴을 더 잘 잡도록 날짜 기반 feature를 추가합니다.

추가할 feature:

| 컬럼명 | 의미 |
| --- | --- |
| `취득학기구분` | 취득월 기준 학기 구분 |
| `신학기준비월여부` | 2월, 8월, 3월, 9월 여부 |
| `예산집행월여부` | 12월 여부 |
| `방학기간여부` | 1~2월, 7~8월 여부 |

학기 구분 예시:

| 월 | 학기구분 |
| --- | --- |
| 3~6월 | 1학기 |
| 7~8월 | 여름 |
| 9~12월 | 2학기 |
| 1~2월 | 겨울 |

주의:

현재 RUL 예측 모델에서는 `취득월`과 취득 시점 feature 중심으로 사용합니다.  
월별 수요 예측 모델에서는 분석 대상 월/학기 feature를 별도로 사용하는 것이 더 적절합니다.

### 3.4.5. 부서별 예산/관리 성향 feature

Phase 1 또는 Phase 2에서 추가한 부서 성향 값을 Phase 4로 전달합니다.

추가할 feature:

| 컬럼명 | 의미 |
| --- | --- |
| `부서예산등급` | LOW/MEDIUM/HIGH |
| `부서예산등급_Code` | 모델용 숫자 코드 |
| `부서교체성향` | 교체가 빠른 부서인지 |
| `부서관리성향` | 점검/관리가 많은 부서인지 |

사용 방식:

- `부서교체성향`은 수명 단축/교체 시점에 약하게 반영
- `부서관리성향`은 점검 이력 증가와 수명 보정에 약하게 반영
- 모델 feature로도 사용

주의:

실제 예산액처럼 보이는 컬럼명은 피합니다.  
예산 금액을 임의로 만들면 근거를 설명하기 어렵기 때문입니다.

## 6. 모델링 측면 기대 효과

### 6.1. RUL/실제수명 예측

기존 모델은 주로 다음 정보로 수명을 예측했습니다.

- 내용연수
- 취득금액
- 운용연차
- 부서가혹도
- 품목/부서 target encoding

보강 후에는 다음 신호가 추가됩니다.

- 실제 사용량
- 수리/점검 누적 이력
- 구매 배치 규모
- 부서별 교체/관리 성향

따라서 단순히 “몇 년 썼는가”가 아니라 “얼마나 많이 쓰였고, 고장 징후가 얼마나 있었는가”를 함께 반영할 수 있습니다.

### 6.2. 월별 수요 예측

월별 수요 예측은 개별 RUL 예측보다 계절성과 묶음 구매 효과가 중요합니다.

보강 후에는 다음 패턴을 더 잘 설명할 수 있습니다.

- 2월/8월 점검 시즌
- 3월/9월 신학기 준비
- 12월 예산 집행
- 동일 배치 장비의 집단 교체
- 특정 부서/품목군의 수요 집중

## 7. 발표 시 설명 문장

발표에서는 다음처럼 설명할 수 있습니다.

```text
초기 합성 데이터는 취득, 운용, 불용, 처분이라는 업무 흐름을 중심으로 생성했습니다.
이후 모델이 단순히 내용연수만 학습하지 않도록,
운영 중 누적되는 점검/수리 이력, 자산별 사용강도, 동일 구매 배치, 학사/예산 시즌, 부서별 관리 성향을 추가했습니다.
단, 실제 불용일자나 처분일자처럼 결과 이후에만 알 수 있는 정보는 feature로 사용하지 않아 데이터 누수를 방지했습니다.
```

## 8. 주의사항

1. 신규 패턴 추가 후에는 Phase 1부터 Phase 4까지 모두 다시 실행해야 합니다.
2. Phase 4 컬럼이 바뀌므로 모델링 노트북도 feature list를 업데이트해야 합니다.
3. 기존 서버가 기대하는 예측 결과 컬럼명은 유지해야 합니다.
4. `model.pkl` 교체 시 서버 입력/출력 스키마가 깨지지 않도록 기존 컬럼은 삭제하지 않습니다.
5. 성능이 좋아져도 feature importance에서 수리/사용강도만 과도하게 높다면 패턴이 너무 강한 것이므로 보정 강도를 낮춰야 합니다.
